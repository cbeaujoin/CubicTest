CubicTest and agile software development

The main reason why CubicTest was created was to fill a gap in the tools that support agile software development (e.g. eXtreme Programming) of web applications. There were few or no tools that let the "customer" (non-technical / business-side person) easily write the tests of a web application, let alone a tool that provided the visual abstractions needed to make the customer comfortable with writing the tests before the application was implemented.

The goal of CubicTest is to make it easy and fun to write web-application tests, for both technical and non-technical people. These tests should ideally be written by the "owner" of the functionality, and preferrably before the application is implemented, such that the tests can be used in all phases of the application under construction.

Benefits of test-driven development of features in a web application

Tests written in CubicTest can be used
  1. as input to the developer when a feature is about to be implemented,
  2. for specifying changes to existing functionality,
  3. to report bugs (represent interaction and input that led to the bug),
  4. to regression test existing features when changes are made, and
  5. for performance tests (future work).

All the above would be in the form of failing tests, and the developer's job would be to make the tests pass.

In line with eXtreme Programming practice, the tests should be written only when they are needed. This is typically in a development iteration where a particular user story is going to be implemented and needs to be detailed. At this time, as much information about the requirements as possible should be contained in the tests.
